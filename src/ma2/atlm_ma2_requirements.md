# Overview

In this project you'll take the domain-adapted model you produced in Mini-Assignment 1 and apply an alignment technique on top of it. The goal is to walk the full post-training pipeline — pretraining → domain adaptation → supervised fine-tuning → preference alignment — and reason about what each stage actually changes in the model's behavior.

As before, the goal is not state-of-the-art results. It is to demonstrate that you understand the alignment pipeline end-to-end: where preference data comes from, why the technique works, what it changes about the model, and how to evaluate it honestly.

---

# Tasks

1. **Start from your Assignment 1 checkpoint**
   Use the domain-adapted model you trained in Mini-Assignment 1 as your starting point. If you used LoRA in Assignment 1, merge the adapter into the base model first (or document why you're keeping it separate). If your Assignment 1 model is unusable for some reason (e.g., severely under-trained or broken), you may swap in the original pretrained checkpoint, but you must justify the decision in your report.

2. **Add a supervised fine-tuning (SFT) stage if needed**
   Alignment techniques like DPO and RLHF assume a starting point that already follows instructions. If your Assignment 1 model is a raw continued-pretraining checkpoint (not instruction-tuned), you'll need a short SFT pass first. You may:
   - Use an existing instruction dataset (Alpaca, Dolly, OpenAssistant, UltraChat, or a domain-relevant equivalent), or
   - Construct a small instruction dataset that reflects your Assignment 1 domain (e.g., Q&A pairs about your chosen domain).
   
   A few hundred to a few thousand examples is enough. Document this stage clearly — it's part of the pipeline.

3. **Choose an alignment technique**
   Pick one and justify the choice given your compute budget and domain:
   - **DPO** (Direct Preference Optimization) — most common, no separate reward model
   - **RLHF with PPO** — classical pipeline; harder to stabilize, more compute
   - **RLAIF** — preferences generated by another LLM acting as judge
   - **ORPO, KTO, SimPO, IPO** — DPO variants with different trade-offs
   - Constitutional AI-style self-critique
   
   DPO or one of its variants is the recommended default for laptop/Colab budgets.

4. **Build or obtain a preference dataset**
   You need pairs of `(prompt, chosen response, rejected response)`. Options:
   - Use an existing dataset: Anthropic/hh-rlhf, argilla/distilabel-intel-orca-dpo-pairs, HuggingFaceH4/ultrafeedback_binarized, nvidia/HelpSteer2.
   - Generate one with RLAIF: have a stronger model (e.g., a GPT-4-class API or a larger open model) score or rank responses from your SFT model.
   - Annotate manually: a few hundred pairs you label yourself — feasible for a focused domain.
   
   You may also combine: a general-purpose preference dataset for broad alignment plus a small domain-specific one. Document size, source, and any cleaning.

5. **Train and evaluate**
   Run the alignment procedure with sensible defaults (TRL's `DPOTrainer` or equivalent). Then perform a three-way comparison:
   - The base pretrained model (before Assignment 1)
   - Your Assignment 1 domain-adapted (+ SFT) model
   - Your aligned model from this assignment

   For each, generate responses to the same fixed set of prompts and compare. Evaluation should include both:
   - **Automatic metrics**: reward model scores, win-rate via LLM-as-judge, perplexity on a held-out set, or task-specific benchmarks if applicable.
   - **Qualitative analysis**: a small set of hand-picked prompts where you discuss specific behavioral changes (e.g., refusal patterns, tone, domain accuracy, hallucination rate, format adherence).

6. **Justify your choices**
   In your report, explain:
   - **Alignment technique** — why this method given your hardware, dataset, and goals? What did you give up by not picking the alternatives?
   - **Preference dataset** — where it came from, why it suits your domain, any quality concerns.
   - **Reference model handling** — for DPO-family methods, what reference policy you used and why (the SFT model is the standard choice, but worth stating).
   - **Hyperparameters** — particularly the beta (or KL coefficient) and learning rate, since alignment is notoriously sensitive to these.

# Deliverables

1. **Reproducible code**
   - A notebook or script set that runs end-to-end, picking up from the Assignment 1 checkpoint.
   - A `requirements.txt` with exact versions (`transformers`, `trl`, `peft`, `torch`, `datasets`, etc.).
   - Random seeds set where appropriate.
   - A `README.md` explaining how to reproduce, including how to point the code at the Assignment 1 checkpoint.

2. **Technical report (max. 5 pages)**
   Include:
   - **Pipeline description**: a clear diagram or narrative of the full pipeline from raw pretrained model → Assignment 1 → SFT (if applicable) → alignment. State what each stage contributes.
   - **Method**: alignment technique chosen, preference dataset, training configuration, hardware.
   - **Results**: the three-way comparison described above. Include training curves (reward margin, loss, KL to reference), automatic metrics, and qualitative examples. Show sample generations side by side from all three model versions on the same prompts.
   - **Limitations and critical discussion**: where did the alignment help? Where did it hurt (e.g., did the model lose domain knowledge from Assignment 1, get more verbose, develop sycophancy, refuse too much)? What does this tell you about the trade-offs of your chosen method? Honest reporting of regressions and surprises is essential — alignment frequently degrades capabilities, and noticing that is part of the assignment.
